## 1. 判定方法

各テストには次のいずれかを記録する。

| 判定 | 意味 |
|---|---|
| PASS | 手順どおり実施し、すべての合格条件を満たした。 |
| FAIL | 再現可能な不具合、基準値超過、表示崩れ、ログ異常のいずれかがある。 |
| BLOCKED | 必要な機器、OS設定、権限、計測手段がなく実施できない。 |
| N/A | 対象環境に機能が存在せず、今回のリリース判定から除外する。理由を必ず残す。 |

推測でPASSにしない。目視項目は可能なら画面録画、性能項目はログまたは計測値、障害項目は復旧前後の状態を証跡として残す。

### 1.1 不具合記録の必須項目

1. テストIDと実施日時
2. Debug／Release、コミットID
3. Windowsのエディション、バージョン、OSビルド
4. モニター解像度、表示倍率、リフレッシュレート、接続方式、主画面
5. タスクバー配置、中央／左揃え、自動的に隠す設定
6. 再現手順、期待結果、実際の結果、再現回数
7. `window-diagnostics.log` の該当時刻
8. 必要に応じてスクリーンショットまたは60fps以上の画面録画

## 2. 試験前の準備

### 2.1 ビルド

リポジトリのルートで次を実行する。

```powershell
dotnet build Windows_SC.slnx -c Debug -p:Platform=x64
dotnet build Windows_SC.slnx -c Release -p:Platform=x64
```

合格条件:

- Debug、Releaseともにビルドエラーが0件。
- 新規警告がない。Releaseの既知のトリミング警告は件数と内容が以前から変化していないことを確認する。
- 実機の機能・性能判定は原則としてx64 Releaseを使う。Debugはログ確認と原因調査に使う。

### 2.2 設定とログの保全

試験前にアプリを終了し、次を別フォルダーへコピーする。

```text
%LOCALAPPDATA%\Windows_SC\settings.json
%LOCALAPPDATA%\Windows_SC\Logs\window-diagnostics.log
%LOCALAPPDATA%\Windows_SC\Logs\window-diagnostics.previous.log
%LOCALAPPDATA%\Windows_SC\Logs\window-diagnostics.detail.log
%LOCALAPPDATA%\Windows_SC\Logs\window-diagnostics.detail.previous.log
```

設定破損試験、項目の大量追加、音声デバイスID変更を行った後は、アプリを終了してから元の `settings.json` を戻す。起動中に設定ファイルを上書きしない。

### 2.3 共通試験データ

機密情報を含まない専用データを用意する。

- 正常なEXE: `notepad.exe`、`calc.exe`
- 正常なコマンド: `DisplaySwitch.exe /extend` など、結果を安全に戻せるもの
- 存在しない対象: 試験専用の架空パス
- 空白とUnicodeを含むフォルダー／ファイル名
- 2台以上の音声出力デバイス。可能なら3台
- 循環コマンド用の2ステップ、3ステップ設定

`DisplaySwitch.exe /internal` は外部画面を切断状態にする。ノートPC内蔵画面がない環境や、遠隔操作中のPCでは実行しない。先に `Win+P` で復旧できることを確認する。

### 2.5 試験中止条件

次の場合は操作を止め、アプリを終了してログを退避する。

- 画面が戻らず操作不能になった。
- 音量が意図せず最大になった、または音声出力が頻繁に切り替わり続ける。
- ランチャーが無限に表示・非表示を繰り返す。
- 常駐中のCPU使用率が継続して1コア相当へ張り付く。
- 同じエラー通知が止まらない。
- Explorer再起動後にタスクバーが復旧しない。

## 3. 実施環境マトリクス

すべての組み合わせを総当たりせず、変更影響に応じて次の優先度で実施する。

| 優先度 | 環境 | 用途 |
|---|---|---|
| P0 | WQHD、125%、下端タスクバー、x64 Release | 基準環境。全機能を実施する。 |
| P0 | 現有の75Hz＋180Hzデュアルディスプレイ | 複数画面・異Hz・実利用回帰。 |
| P1 | 60Hz、100%または125%、単一画面 | 低リフレッシュレートのモーション確認。 |
| P1 | 120Hzまたは144Hz、異なるDPIの複数画面 | 高Hz・DPI境界・負座標確認。 |
| P1 | ライト／ダーク、透明効果とアニメーション効果のオン／オフ | 外観とアクセシビリティ。 |
| P2 | 150／200／300%、4K、縦置き、1～4画面 | 拡張配置試験。 |
| P2 | RDP、Home／Pro／Enterprise、表示言語差 | 参考互換性試験。 |

設計上の基準点は60／120／144Hz、100／125／150／200／300%である。機器がない項目はFAILではなくBLOCKEDとし、75／180Hzの結果で実施済みに置き換えない。

## 6. スタート連動・手動表示・終了

各項目は通常速度で10回、その後約300ms間隔の連続操作を10回行う。

| ID | テスト項目 | 方法 | 合格条件 | 注意点 |
|---|---|---|---|---|
| MOT-01 | Windowsキー起動 | Windowsキー単体でスタートを開く。 | スタート本来の動作を妨げず、ランチャーがスタート右側へ下から表示される。スタート検索が入力可能。 | `Win+E`等との違いも確認する。 |
| MOT-02 | Windowsキー終了 | スタート連動表示中にWindowsキー単体を押す。 | 約125msの退出が即時開始し、1秒近い待ちや取り残しがない。 | OSの非表示検出待ちへ戻っていないこと。 |
| MOT-03 | スタートボタン | タスクバーのスタートボタンをクリックして開閉する。 | キー操作と同じ位置・方向で表示され、閉じると退出する。 | タスクバー中央／左揃えの両方で実施。 |
| MOT-04 | 外側クリック | ランチャーを一度操作後、画面外をクリックする。 | クリック先の操作を妨げず、約125msで退出する。 | 透明な領域が他アプリのクリックを遮らないこと。 |
| MOT-05 | Escape | ランチャー操作中にEscapeを押す。 | 1回の退出だけが発生し、スタートや他アプリへ不要なキー入力を送らない。 | XAMLとWin32の二重処理が見えないこと。 |
| MOT-06 | 手動ホットキー | `Ctrl+Alt+Space`で表示／非表示を繰り返す。表示後はランチャーへ一度も触れず、外部ウィンドウもクリックする。 | スタート状態と無関係にトグルし、多重表示しない。手動表示直後からフォーカスを取得し、最初の画面外クリックでクリック先を妨げず退出する。 | 他アプリとのショートカット競合と、`activation-state`ログを確認する。 |
| MOT-07 | Windowsショートカット | `Win+E`、`Win+R`、`Win+I`、`Win+Shift+S`を使う。 | 対象アプリだけが起動し、Windows_SCは誤表示しない。 | Windowsキーを離す順序も変える。 |
| MOT-08 | 進入中の終了 | 進入開始直後にEscape、Windowsキー、外側クリックを行う。 | 点滅、位置飛び、古い完了通知による再表示がない。最終状態がHidden。 | 60fps以上の録画を推奨。 |
| MOT-09 | 退出中の再表示 | 退出開始直後にWindowsキー、ホットキー、トレイ表示を使う。 | 現在位置から表示へ反転し、座標リセット、瞬間移動、二重アニメーションがない。 | 100回反復は安定性試験で行う。 |
| MOT-10 | 検索・フォーカス共存 | スタート検索へ連続入力し、キーボードで候補を移動する。 | ランチャーがフォーカスを奪わず、検索入力と候補操作が継続できる。 | SearchHost誤検出にも注意する。 |

### 6.1 モーションの合格基準

- 進入は約250ms、退出は約125ms。アニメーション無効時は即時遷移する。
- ランチャー面、背景、影、カードが一体で動き、カード単位の遅延や終端の段付きがない。
- バウンド、オーバーシュート、拡大縮小、横方向の位置飛びがない。
- 60／120／144Hz、および75／180Hzの実機で目視できる停止や周期的なカクつきがない。
- スタート確認から表示要求まで通常50ms以内。
- Windowsキーから最初の表示要求まで、監視準備済みの通常状態で100～150msを目標とする。

| DSP-11 | スリープ復帰 | 表示を閉じてスリープし、復帰後に各入口を試す。 | 監視、トレイ、配置、音声状態が自動復旧する。CPU高止まりがない。 | USB音声の再接続完了を待って音声試験する。 |

## 15. 性能・安定性

### 15.1 計測方法

1. x64 Releaseを使用する。
2. 起動後2分待ち、音声デバイスとUI Automationの初期準備を完了させる。
3. タスクマネージャーまたはPerformance MonitorでCPU、ワーキングセット、GPUを記録する。
4. Windowsキー起動を30回行い、ログの`LaunchTiming`と`Motion`を集計する。
5. 最小、中央値、95パーセンタイル、最大を記録する。初回コールド値は分ける。
6. 録画ソフト、デバッガー、ウイルススキャン等の計測影響を記録する。

| ID | テスト項目 | 方法 | 合格条件 | 注意点 |
|---|---|---|---|---|
| PERF-01 | アイドルCPU | 非表示で5分間計測する。 | 通常ほぼ0%。周期的なスパイクや継続ポーリングがない。 | 瞬間値だけでFAILにしない。平均と原因を記録する。 |
| PERF-02 | アイドルメモリ | 起動2分後と30分後のワーキングセットを記録する。 | 500MB以下で、操作なしに継続増加しない。 | OS共有メモリとPrivate Bytesを混同しない。 |
| PERF-03 | 表示時間 | Windowsキー起動30回のログを集計する。 | 通常の`start-to-request-ms`が50ms以内。`key-to-start-ms`は通常10～100ms。コールド値は別枠。 | スタートがOS側で開かなかった回を除外し、理由を記録する。 |
| PERF-04 | モーション時間 | Entrance／Exitの`elapsed-ms`を30回集計する。 | Entrance約250ms、Exit約125ms。Exitが200ms超で継続しない。 | 完了通知のログ時刻で判定する。 |
| PERF-05 | UIA負荷 | 非表示5分、Windowsキー後1.5秒、表示中を比較する。 | 非表示中に連続スキャンせず、`scan=slow`が連続しない。1秒級COM待機が通常操作を止めない。 | 詳細ログの追加自体で負荷を変えない。 |
| PERF-06 | 途中反転100回 | 表示・退出・再表示を100回繰り返す。 | クラッシュ、点滅、位置飛び、状態取り残し、意図しない再表示が0回。 | 自動キー送信でOS操作を妨げない。手動または安全な試験環境で行う。 |
| PERF-07 | 8時間常駐 | 通常利用または制御された反復を含め8時間以上常駐させる。 | CPU高止まり、継続的メモリ増加、ハンドル増加、ログ洪水、操作不能がない。 | 開始・1h・4h・8hの値を記録する。 |
| PERF-08 | 数千回回帰 | 専用環境で表示・非表示を数千回行う。 | プロセスが継続し、GDI／USER／ハンドル数が単調増加しない。 | 個人の作業PC上で無人入力を行わない。 |

### 15.2 ログ合格条件

ログ: `%LOCALAPPDATA%\Windows_SC\Logs\window-diagnostics.log`

- 通常のWindowsキー起動で`start-to-request-ms <= 50`。
- Entranceは約250ms、Exitは約125ms。
- 通常のスタート表示で`start-confirmation=expired`が継続発生しない。
- `scan=slow`が連続しない。
- アクション成功直後に意図しない`Exiting → EnteringWithStart`がない。
- トレイ起動時にadd success、終了時にremoveが記録される。
- 同一操作についてshow、exit、actionが重複していない。
- ログが2MBを超えた状態で次回起動した場合、previousへローテーションし、アプリが停止しない。

## 18. リリース合格条件

次をすべて満たした時点でPhase 5を完了とする。

- P0環境のAPP、TRAY、MOT、DSP、SET、ACT、AUD、CYC、UI、RECがすべてPASS。
- 対応外タスクバー、対応範囲内の表示領域不足、UI Automation失敗、モニター切断、DPI変更がクラッシュせず設計どおり処理される。
- 75Hz／180Hzデュアル環境で配置、モーション、音声、トレイの回帰がない。
- 60Hzと120Hz以上の環境でモーション停止、位置飛び、終端の段付きがない。
- 通常の`start-to-request-ms`が50ms以内で、Exitが通常約125ms。
- 重大度が高い既知不具合、データ破損、終了不能、操作不能が0件。

次をすべて満たした時点でPhase 6と正式な安定版の検証を完了とする。知人向け試験版は、簡易テストチェックリストの配布条件を使用できる。

- Phase 5合格条件を維持している。
- PERF-06の途中反転100回とPERF-07の8時間常駐がPASS。
- アイドルメモリ500MB以下で、CPU、メモリ、USER／GDI／ハンドルに継続増加がない。
- Explorer再起動、スリープ復帰、モニター構成変更後に自動復旧する。
- x64 Releaseをクリーン環境で導入、起動、自動起動、終了、削除できる。
- BLOCKEDとN/Aに理由があり、未実施リスクを配布メモへ記載している。

## 20. 推奨実施順

1. 設定とログを退避する。
2. Debug／Releaseビルドを確認する。
3. P0基準環境でAPP、TRAY、MOTのスモーク試験を行う。
4. SET、ACT、AUD、CYC、UIを行う。
5. 75Hz／180Hzデュアル環境でDSPとモーション回帰を行う。
6. 障害を安全に戻せることを確認してRECを行う。
7. 60Hz、120／144Hz、主要DPI、外観・アクセシビリティを行う。
8. 機能不具合を解消してから性能計測と長時間試験を行う。
9. 最後にクリーン環境のパッケージ試験を行う。
10. FAIL、BLOCKED、N/Aとログをレビューし、Phase 5／6の合否を確定する。
