# Windows_SC バージョン管理

現在の開発版は `0.5.4-dev` とする。Phase 5で初回配布前の検証を進めていることを表す暫定バージョンである。

## 更新方法

通常はリポジトリ直下の `Version.props` にある次の2項目だけを更新する。

- `VersionPrefix`: 公開するバージョン番号。`メジャー.マイナー.パッチ`形式とする。
- `VersionSuffix`: 開発版では`dev`、正式版では値を空にする。

設定画面からコピーする環境情報には、この値が自動的に表示される。実行ファイルのアセンブリバージョンとファイルバージョンにも`VersionPrefix`が反映される。

利用者に影響する変更は、次のバージョンを公開するまで`CHANGELOG.md`の「未リリース」へ追記する。公開時にその内容を新しいバージョン見出しへ移し、公開日を記録する。実装途中の細かな変更はバージョン番号を上げず、Gitのコミット履歴で追跡する。

## 番号の目安

- パッチ: 不具合修正だけを行った場合（例: `0.5.0` → `0.5.1`）
- マイナー: 後方互換性のある機能追加（例: `0.5.0` → `0.6.0`）
- メジャー: 設定互換性を壊す変更や正式版の大きな区切り

## 開発段階の接尾辞

- `dev`: 日常の開発・検証中
- `beta.1`, `beta.2`: 主な機能が揃い、複数環境で試験する版
- `rc.1`, `rc.2`: 正式版候補。原則として不具合修正だけを行う版
- 接尾辞なし: 正式に配布する安定版

番号はコミットごとではなく、配布物を作る区切りで更新する。同じバージョンの配布物を作り直す必要がある場合は、内容を判別できるよう最低でもパッチ番号を上げる。

初回正式配布時は`VersionPrefix`を`1.0.0`にし、`VersionSuffix`を空にする。MSIXを使う場合は、数値4区切りが必要な`Package.appxmanifest`の`Identity Version`も同じリリース番号へ合わせる。
