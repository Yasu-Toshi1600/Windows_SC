## 2. 製品概要

Windows 11のスタートメニューと同時に呼び出せる、常駐型の軽量ランチャーを提供する。アプリ、ファイル、URL、コマンド、音声出力先、マスター音量を、ボタン、循環切り替え、スライダーから操作できる。

主な利用者は本人と知人を想定し、一般公開サービス、企業向け集中管理、設定のオンライン同期は対象外とする。

## 3. 対応範囲

### 対応対象

- Windows 11 バージョン25H2以降
- x64
- 日本語環境
- 下端タスクバー
- タスクバーの中央揃え／左揃え
- 720p以上で、スタートとランチャーを配置できる作業領域
- 複数モニター、異なるDPI、負座標を含む構成

製品として配布・判定するのはx64自己完結版だけとする。プロジェクトに残るx86／ARM64定義は製品サポートを意味しない。

### 対象外

- Windows 10、Windows 11 24H2以前
- x86、ARM64の配布保証
- 上、左、右に配置したタスクバー
- 720p未満向けの可変高さ・専用レイアウト
- Microsoft Store公開、自動更新、クラウド同期
- 複数ページ、フォルダー、プラグイン、ウィジェット
- コマンドの終了待機、終了コード、タイムアウト、標準出力取得

起動時にOSビルド、OS／プロセスアーキテクチャ、タスクバー位置を検査する。Windows 11 25H2（ビルド26200）未満、x64以外、下端以外のタスクバー、またはタスクバー位置を確認できない環境では、理由と対応環境をエラーダイアログで表示してアプリを自動終了する。日本語環境と解像度は起動時の強制終了条件に含めない。

## 4. アプリのライフサイクル

### 起動

1. 保存場所と通常ログを初期化し、単一インスタンスを取得する。既に起動中なら2個目は常駐せず終了する。
2. 対応環境を検査し、非対応ならエラーダイアログを表示して終了する。
3. 設定を読み、破損または未対応なら退避して既定設定へ復旧する。
4. ランチャーウィンドウ、入力監視、スタート監視、音声サービス、通知領域アイコンを初期化する。
5. ランチャーと設定画面を表示せず、通知領域に常駐する。

UI Automationのイベント登録は外部プロバイダーで遅れる場合があるため、スキャン用スレッドと分離する。監視準備中でもトレイ、設定、終了、手動ホットキーを利用可能にする。

### 表示入口

- Windowsキー単体
- タスクバーのスタートボタンクリック
- `Ctrl+Alt+Space`
- 通知領域メニューの「ランチャーを表示」

Windowsキーを含む別ショートカットは単体入力として扱わない。`Ctrl+Alt+Space`は現行固定値であり、変更UIは初回版に含まれない。

### 終了

- 通知領域メニューの「終了」
- 設定画面の終了操作

終了時は入力フック、UI Automation監視、トレイアイコン、音声サービス、ログライター等を解除する。ランチャーを閉じる操作はアプリ終了ではなく、非表示常駐への復帰である。

## 5. ランチャー画面

### ウィンドウ

- 枠、タイトルバー、リサイズ、最大化、最小化を持たない。
- タスクバーとAlt+Tabの一覧に表示しない。
- 表示中は常に手前に置く。
- 高さは現行600 physical pixelを基準とし、項目部分を縦スクロールする。
- スタート連動時は検出したスタートと同じモニターの右側へ配置する。
- 手動表示時はカーソルがあるモニターの作業領域中央へ配置する。
- スマートフォン連携パネルは設定値により予約幅を確保する。個別パネルのUI Automation走査は性能上無効である。

右側の幅が不足する場合は、標準またはコンパクトの列数を減らす。それでも最低幅を確保できない場合は表示を中止し、理由をログへ残す。

### レイアウト

- 標準: 通常ボタンを最大2列
- コンパクト: 通常ボタンを最大4列
- 循環切り替えと音量スライダー: 通常ボタン2列分
- 利用可能幅に応じて列数を減らす
- 表示するたびスクロール位置を先頭へ戻す

### 表示・非表示

- スタート連動表示ではランチャーをアクティブ化せず、スタート検索のフォーカスを奪わない。
- 表示要求時にランチャーが別の仮想デスクトップへ所属している場合は、現在操作中の仮想デスクトップへ移動し、古い表示状態を破棄してから配置・表示する。
- ランチャーへフォーカスまたはポインター操作が移ると対話状態になり、スタートが閉じても表示を維持する。
- Windowsキー、スタートの終了、外側クリック、Escape、設定表示、成功後に閉じる操作で退出する。
- 退出中の新しい表示要求は現在位置から進入へ反転する。

詳細な状態と時間はモーション仕様書、検出条件はスタート保守ガイドに従う。

## 6. ランチャー項目

### 共通

各項目はID、種類、表示名、有効状態、配置順を持つ。現行の種類はボタン、循環切り替え、音量スライダーである。

### ボタン

次のいずれかを起動する。

- アプリ、実行ファイル
- ファイル、フォルダー、URL、Windows URI
- コマンド、bat、cmd

設定可能な値:

- 起動対象またはコマンド
- 引数
- 作業フォルダー
- コマンド画面を非表示にするか。ただしコマンド／bat実行時だけ表示する。

起動対象と作業フォルダーでは環境変数を展開する。PATHやApp Pathsで解決できるEXEはフルパスなしで起動できるが、Storeアプリ等はURIや解決可能な対象を指定する。

ボタンは起動要求に成功したらランチャーを閉じる。「実行後」の選択欄は表示しない。失敗時はランチャーを残してエラーを表示する。

### 循環切り替え

2件以上を登録し、操作ごとに登録順で次へ進み、末尾の次は先頭へ戻る。

#### 音声出力

- 登録した既定出力デバイスを順番に切り替える。
- 切断、無効、利用不能なデバイスは飛ばす。
- 再接続後は再び候補へ戻す。
- 現在の既定デバイス以外に切り替え可能な候補がない場合は操作不能にする。
- 実行位置は現在の既定出力を基準に決め、永続化しない。

#### コマンド

- 表示名、対象、引数、作業フォルダー、画面表示方針をステップごとに持つ。
- 起動要求の成否にかかわらず、操作するたび登録順で次のステップへ進む。
- 起動要求に失敗した場合はランチャーを閉じず、エラーを表示する。
- アプリはプロセス終了と終了コードを追跡しないため、コマンド自身の失敗は判定しない。

循環切り替えは「成功後に閉じる」または「表示したまま」を設定でき、既定値は「成功後に閉じる」とする。
「表示したまま」で表示ありのコマンドを起動した場合、コマンド画面へのフォーカス移動ではランチャーを閉じない。ランチャーを再度アクティブ化した後は、通常どおり画面外クリックで閉じる。

### 音量スライダー

- 現在の既定出力デバイスのマスター音量を操作する。
- アプリ別音量は対象外とする。
- 既定デバイスが利用不能な場合は操作不能にする。
- 実行後動作は設定しない。

## 7. 設定画面

単一の設定画面で、一般設定、項目一覧、選択中項目の編集、トラブルシューティングを提供する。

### 一般設定

- スマートフォン連携パネルの予約
- 標準／コンパクトレイアウト
- Windowsログオン時の自動起動

### 項目編集

- 追加、削除、上下移動
- 種類、表示名、有効状態
- ボタンの実行種類と対象
- 循環切り替えのモード、音声デバイスまたはコマンドステップ
- 循環切り替えの実行後動作

上下移動後も対象を選択したまま表示領域へ追従させる。選択中の実行種類を表示し、種類に応じて入力名、例、補助設定を切り替える。

### 保存

- 自動保存しない。
- 下部の「保存」で入力検証、JSON保存、自動起動反映、ランチャー再構築を行う。
- 保存前の変更をランチャーや終了時保存へ混入させない。
- 詳細診断ログだけはトラブルシューティング内の専用適用を使う。
- 全列挙値と種類ごとの必須データを読込時に検証し、不正な設定は破損バックアップへ退避する。
- 保存はJSONを先に確定してから自動起動を反映する。自動起動が失敗した場合は両方を保存前へ戻し、実際の自動起動状態を利用者へ表示する。

新規作成または破損復旧時の既定項目は、メモ帳、電卓、エクスプローラー、Windows設定とする。

## 8. 通知領域

起動中は`Windows_SC`アイコンを1個表示する。左クリック、右クリック、キーボード選択から同じメニューを開く。

- ランチャーを表示
- 設定
- 終了

「非表示」は設けない。メニュー操作でフォーカスが移り、必要な場合はランチャー側のlight-dismiss処理が働く。Explorer再起動後はタスクバー再生成通知を受け、アイコンを再登録する。

## 9. 現行アーキテクチャ

| 領域 | 主な実装 | 責務 |
|---|---|---|
| 起動と寿命 | `App.xaml.cs` | サービス生成、単一インスタンス、設定読込、トレイ、設定画面、終了 |
| ランチャーUI | `MainWindow.xaml(.cs)` | 表示入口、配置、状態同期、モーション開始、light-dismiss |
| ランチャー状態 | `MainWindowViewModel`、`LauncherItemViewModel` | 項目構築、実行、音声状態反映 |
| 設定UI | `SettingsWindow.xaml(.cs)`、`SettingsViewModel` | 編集用コピー、検証、明示保存、診断操作 |
| スタート監視 | `HybridStartMenuMonitor`、`UiAutomationStartMenuInspector`、`StartMenuWindowInspector` | UIA／Win32検出、Snapshot、フォールバック |
| モーション | `LauncherMotionCoordinator`、`CompositionLauncherMotionService` | 論理状態、Translation、Opacity、反転、完了通知 |
| 配置 | `LauncherPlacementService` | モニター、DPI、作業領域、列数、ランチャー矩形 |
| 入力 | `GlobalInputService`、`GlobalWindowsKeyMonitor`、`WindowInteropService` | Windowsキー、ホットキー、外側クリック、Escape |
| アクション | `ActionExecutionService` | Shell／cmdによる起動要求 |
| 音声 | `WindowsAudioOutputService` | デバイスキャッシュ、既定出力切り替え、マスター音量 |
| 設定保存 | `JsonSettingsRepository`、`LauncherSettingsValidator`、`ApplicationDataPaths` | JSON、移行、バックアップ、検証、保存先 |
| 診断 | `DiagnosticLogger`、`LogPrivacySanitizer`、`ApplicationInformation`、`EnvironmentInformationService` | 通常／詳細ログ、匿名化、環境情報の取得・変更検出 |
| OS統合 | `WindowsSystemTrayService`、`RegistryStartupService`、`SingleInstanceService` | トレイ、自動起動、二重起動防止 |

Win32、COM、WPF UI Automation等のOS依存処理はサービス境界へ置き、ViewModelから直接呼ばないことを基本とする。例外的な現行コードを追加するときは、将来の分離方針を残件へ記録する。

## 10. 主なデータフロー

### スタート連動表示

```text
Windowsキー／スタートクリック
  → 入力・スタート監視
  → StartMenuSnapshot
  → MainWindowの状態同期
  → LauncherPlacementService
  → AppWindowを最終位置へ表示
  → Composition進入
```

### 項目実行

```text
項目操作
  → LauncherItemViewModel
  → ActionExecutionService または WindowsAudioOutputService
  → 成功／失敗
  → 循環位置・表示状態更新
  → 必要な場合だけランチャー退出
```

### 設定保存

```text
設定画面の編集コピー
  → 入力検証
  → JSON保存と自動起動反映
  → MainWindowViewModelへ適用
  → ランチャー項目再構築
```

## 11. 設定・ログ保存

```text
%LOCALAPPDATA%\Windows_SC\
├─ Settings\
│  ├─ settings.json
│  └─ Backup\
│     └─ settings.corrupt-日時.json
└─ Logs\
   ├─ window-diagnostics.log
   ├─ window-diagnostics.previous.log
   ├─ window-diagnostics.detail.log
   └─ window-diagnostics.detail.previous.log
```

- MSIX／非パッケージ版とも利用者から見える保存先を統一する。
- 自動起動は`HKCU\Software\Microsoft\Windows\CurrentVersion\Run`へ登録する。MSIX版ではこのキーだけをレジストリ仮想化の対象外とし、タスクマネージャーとWindowsログオン処理から参照できる実レジストリへ書き込む。登録・解除後は同じ値を読み戻して反映を確認する。
- 旧保存場所は初回読込時に新構成へ移行する。
- 設定はスキーマバージョンを持つJSONとする。
- 不正な設定は日時付きで退避し、既定設定へ復旧する。
- 通常ログは常時有効、詳細ログは専用操作から24時間だけ有効とする。
- アプリは設定とログを外部へ自動送信しない。
- 設定画面から開くフォルダーはデータルートだけとし、`Settings`と`Logs`はその配下で確認する。
- 環境情報は起動時に通常ログへ記録する。表示環境の変更通知は750ms待ってまとめ、前回記録した内容から変化した場合だけ再記録する。

UI Automation要素名は取得・記録しない。詳細ログにはコマンド、音声デバイス名、例外情報等が含まれる可能性があるため、共有前に利用者が確認する。トラブルシューティング画面にもこの注意を常時表示する。

## 12. エラー処理と安全性

- 対応環境検査の不一致は、短い日本語エラーダイアログと通常ログを残して起動を中止する。それ以外の予期可能な設定、音声、OS APIの失敗は可能な限りアプリ全体を終了させない。
- 設定破損では元ファイルを退避し、黙って上書きしない。
- アクション失敗ではランチャーを維持し、再操作可能にする。
- コマンド欄は`cmd.exe /d /s /c`で実行され、シェルメタ文字を解釈する。信頼できる利用者が入力する前提とし、外部から取得した設定を無確認で実行しない。
- 通常機能で管理者権限を要求しない。対象側が昇格を要求する場合はOSのUAC動作に従う。
- ログへ秘密値を意図的に記録しない。ただし詳細ログは完全匿名ではない。
- UIスレッド、AppDomain、未監視Taskの例外境界で、例外種別、HRESULT、終了有無を通常ログへ記録する。致命的例外は処理済みにせず、記録後も通常の異常終了動作に従う。

## 13. ビルド、配布、バージョン

- Target Framework: `net8.0-windows10.0.19041.0`
- Windows App SDK: 1.8
- 配布アーキテクチャ: x64
- Windows App SDK自己完結: 有効
- Release ReadyToRun: 有効
- トリミング: 無効

## 15. 設計変更時のルール

1. 現行動作を変更する場合は本書または該当する専門仕様を先に更新する。
2. スタート検出とモーションを変更する場合は、両文書の責務境界を保ち、片方だけに矛盾する状態遷移を追加しない。
3. 設定モデル変更ではスキーマ互換性、既定値、破損復旧、旧版移行を同時に検討する。
4. 新しいログ項目は通常／詳細の分類と個人情報リスクを確認する。
5. 利用者影響のある変更は`CHANGELOG.md`へ追記する。
6. 配布前はVersion、コミット、成果物、ログの版が一致することを確認する。
