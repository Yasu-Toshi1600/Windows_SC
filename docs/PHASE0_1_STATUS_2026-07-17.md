# Phase 0-1 実装・検証まとめ

> この資料は2026-07-17時点の検証記録であり、監視間隔や未実装事項には古い情報を含む。現行仕様は[スタート連動機能 保守ガイド](START_MENU_INTEGRATION_MAINTENANCE.md)を参照する。

作成日: 2026-07-17  
状態: 技術検証中

## 1. 今日実装した内容

### ランチャーウィンドウ

- WinUI 3 の枠なしウィンドウを作成した。
- タイトルバー、リサイズ、最大化、最小化を無効化した。
- タスクバーおよび Alt+Tab の一覧に表示しない。
- 常に最前面に表示する。
- 現在の暫定サイズは 500 x 600 px で、プライマリモニター中央に配置する。
- 起動直後は表示せず、バックグラウンドで入力を待機する。
- Escape キーで非表示にする。

### 代替グローバルホットキー

- `Ctrl+Alt+Space` を `RegisterHotKey` で登録した。
- ホットキーを押すたび、ランチャーの表示／非表示を切り替える。
- 表示時はランチャーをアクティブ化する。

### Windows キー単体の検出

- `WH_KEYBOARD_LL` による低レベルキーボード監視を実装した。
- 左右両方の Windows キーに対応した。
- Windows キーを押している間に別キーが押された場合は、Windows ショートカットとしてランチャー処理を行わない。
- Ctrl、Alt、Shift を先に押している場合も単体押下として扱わない。
- 注入された疑似キー入力は無視する。
- Windows キー単体を離してから 180 ms 後に診断と表示処理を行う。

### 自動非表示

- ランチャー表示後 750 ms は自動非表示判定を保留する。
- 表示中は 100 ms 間隔で次の状態を確認する。
  - ランチャー自身にフォーカスがあるか
  - スタートメニューを検出できているか
- 両方とも false の場合はランチャーを非表示にする。
- Windows キー経由では、スタートからフォーカスを奪わないよう非アクティブ表示する。
- `Ctrl+Alt+Space` 経由ではアクティブ表示する。

### 診断ログ

- セッション開始、入力監視、Windows キーの単体／ショートカット判定を記録する。
- ランチャーの表示・非表示と、その理由を記録する。
- ランチャーのフォーカスとスタート検出状態が変化した際に記録する。
- Win32 ウィンドウ候補の HWND、プロセス、クラス、矩形を記録する。
- UI Automation 要素のプロセス、名前、AutomationId、ControlType、HWND、矩形を記録する。
- ログは最大 2 MB とし、超過時は `window-diagnostics.previous.log` へローテーションする。
- 自己完結 EXE で確認したログ出力先:

```text
%LOCALAPPDATA%\Windows_SC\Logs\window-diagnostics.log
```

- MSIX 実行時はファイル仮想化により、次のような `LocalCache` 配下へ保存される場合がある。

```text
%LOCALAPPDATA%\Packages\<PackageFamilyName>\LocalCache\Local\Windows_SC\Logs\
```

### スタートメニュー検出

#### Win32 方式

- `EnumWindows`、`GetWindowRect`、前面ウィンドウ、DWM クローク状態を使う方式を実装した。
- 対象候補は `StartMenuExperienceHost`、`ShellExperienceHost`、`PhoneExperienceHost`。
- Windows 11 25H2 実機では候補が 0 件となり、この方式だけでは検出できなかった。

#### UI Automation 方式

- `System.Windows.Automation` を使用するため、Windows Desktop WPF Framework Reference を追加した。WPF の画面は使用せず、UI Automation クライアントアセンブリのみ利用する。
- UI Automation 処理は WinUI の UI スレッドから分離し、専用 STA スレッドで実行する。
- 検証中は 200 ms 間隔でデスクトップ UI Automation ツリーを確認する。
- デスクトップ直下要素、フォーカス要素、親要素の矩形をログへ記録する。
- 25H2 実機では、スタートを開いた直後のフォーカス要素が次として取得できた。

```text
process=SearchHost
automation-id=SearchTextBox
control-type=ControlType.Edit
```

- その親要素から、次のスタート UI 候補矩形を取得できた。

```text
rect=(0,267)-(1073,1380)
size=1073x1113
```

- `StartMenuExperienceHost` と `SearchHost` を検出対象としている。

### Windows App SDK

- Windows App SDK 2.2.0 から公式安定版 1.8.6 (`1.8.260317003`) へ変更した。
- `WindowsAppSDKSelfContained=true` を設定し、自己完結型を有効化した。
- Windows App Runtime を別途インストールしていない環境でも、ビルド出力の EXE が起動することを確認した。
- x64 Debug ビルドは警告 0 件、エラー 0 件。

## 2. 現在の実際の挙動

### Windows キー

現在は完全なスタート同期ではない。

1. Windows キー単体を検出する。
2. 180 ms 後にスタート関連要素を診断する。
3. ランチャーの表示状態を単純に反転する。
4. 表示後は、UI Automation／Win32 のスタート検出とフォーカス状態により自動非表示を判定する。

そのため、「スタートが開いたから表示」「スタートが閉じたから非表示」という状態機械にはまだなっていない。

### `Ctrl+Alt+Space`

- スタートとは独立した手動トグル。
- 非表示時はアクティブ表示し、表示中は即座に非表示にする。

### スタート表示中

- `SearchHost/SearchTextBox` がフォーカスされている場合、UI Automation でスタート表示を検出できる。
- スタート内でフォーカスが別要素へ移った場合や、マウスだけで操作した場合の検出精度は未検証。

## 3. 判明した問題と制約

1. Win32 のトップレベルウィンドウ列挙では、25H2 のスタートメニューを取得できない。
2. UI Automation ではスタートが `SearchHost` として公開される場合があり、`StartMenuExperienceHost` だけでは不足する。
3. `SearchHost` はタスクバー検索等でも使用される可能性があり、`SearchTextBox` だけでスタートと断定すると誤検出する可能性がある。
4. 現在の UI Automation 監視は 200 ms ポーリングであり、最終的な軽量化要件を満たす方式ではない。
5. スマートフォン連携パネルの UI Automation 要素と個別矩形は、まだ特定できていない。
6. Windows キー処理が依然として `ToggleWindow` を使用しており、スタート状態との同期が不完全。
7. 複数モニター上のスタート表示モニター判定は未実装。
8. PowerShell 5 の既定文字コードで UTF-8 ログを読むと、日本語が文字化けして見える場合がある。ログ自体の文字コードとログ閲覧方法を統一する必要がある。
9. MSIX と自己完結 EXE でログの実保存場所が変わるため、設定画面からログフォルダーを開く機能が必要。

## 4. 次回の優先作業

### 優先度 1: スタート連動の状態機械

Windows キーによる単純トグルを廃止し、次の状態を管理する。

```text
Hidden
StartOpening
VisibleWithStart
VisibleInteractive
StartClosing
```

- Windows キー単体は監視開始のきっかけにのみ使用する。
- UI Automation がスタート表示を確認した時点でランチャーを表示する。
- スタートが閉じ、ランチャーにもフォーカスがなければ非表示にする。
- ランチャー操作中は、スタートが閉じても表示を維持する。
- Windows キーショートカットでは状態遷移を開始しない。

### 優先度 2: スタート識別条件の強化

- `SearchHost/SearchTextBox` の親ツリーにある名前、AutomationId、矩形、画面下端との位置関係を採取する。
- スタートをマウスで操作、検索、すべてのアプリ表示、フォーカス移動した場合を記録する。
- タスクバー検索を開いた場合のツリーを比較し、誤検出を防ぐ。
- 中央揃え／左揃えで比較する。

### 優先度 3: スマートフォン連携パネル

- スマートフォン連携パネル表示あり／なしの UI Automation ツリー差分を採取する。
- `PhoneExperienceHost`、`SearchHost`、スタート親要素配下を比較する。
- パネルの個別矩形を特定できるか検証する。
- 個別矩形が取得できない場合は、スタート複合ウィンドウ全体の右端を回避境界として使う。

### 優先度 4: イベント駆動化

- UI Automation のフォーカス変更イベント、構造変更イベントを検討する。
- `SetWinEventHook` の前面／表示／非表示イベントを補助的に使用する。
- ポーリングはイベント欠落時の短時間フォールバックに限定する。
- アイドル CPU とメモリを計測する。

### 優先度 5: 複数モニターと配置

- 取得したスタート矩形が属するモニターを判定する。
- スタートメニュー右端、スマートフォン連携パネル右端を基準にランチャーを配置する。
- 必要な約 400 effective pixel を確保できない場合は表示せずエラーにする。
- 下端以外のタスクバー配置ではエラー後に終了する。

### 優先度 6: 診断機能の整理

- 設定画面または通知領域からログフォルダーを開けるようにする。
- ログ文字コードと閲覧方法を統一する。
- 通常ログと詳細 UI Automation ログを分ける。
- 詳細診断を設定で有効／無効にできるようにする。

## 5. Phase 0-1 の完了条件

- Windows キー単体で、スタートが実際に開いた場合だけランチャーを表示する。
- Windows キーショートカットではランチャーを表示しない。
- スタート表示中は、ランチャーが非アクティブでも表示を維持する。
- スタートが閉じ、ランチャーも非アクティブなら自動で非表示にする。
- `Ctrl+Alt+Space` は独立した代替操作として安定して動作する。
- スタート矩形と表示モニターを取得できる。
- スマートフォン連携パネルの有無と回避境界を決定できる。
- 定常監視がイベント駆動中心で、アイドル負荷が十分に低い。
- Windows 11 25H2 x64 で繰り返し操作しても誤表示・点滅・即時非表示を起こさない。
