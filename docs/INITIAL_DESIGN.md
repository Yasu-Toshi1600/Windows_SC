## 0. 前提と用語

- 対象 OS は Windows 11 バージョン 25H2 以降、x64 のみとする。Windows 10、Windows 11 24H2 以前、ARM64、x86 は対象外とする。
- 技術構成は C#、.NET 8 以降、WinUI 3、MVVM とする。
- 本文では、スタートメニューの横に出す常駐 UI を「ランチャー」、編集用 UI を「設定画面」と呼ぶ。
- 「モバイルデバイスの表示」は、Windows 11 のスタートメニュー横に表示されるスマートフォン連携パネルを指す。
- 現在のプロジェクトは `net8.0-windows10.0.19041.0`、Windows App SDK 1.8、WinUI 3、x86/x64/ARM64、MSIX、自己完結配布対応の構成である。製品としての検証・配布対象は x64 のみに限定する。ReleaseではReadyToRunを有効にする一方、JSON、COM音声制御、WPF UI Automationの実行時互換性を優先してトリミングを無効にする。
- ウィジェット（温度など）は将来拡張として境界だけ設計し、初回リリースには含めない。

## 1. 要件定義

### 1.1 目的

Windows 11 のスタートメニューと同時に呼び出せる、Stream Deck 型の軽量ランチャーを提供する。アプリ、ファイル、URL、コマンド、Windows 機能などをボタンやスイッチ等から素早く操作でき、利用者自身が設定画面で内容を編集できることを目的とする。

### 1.2 機能要件

#### 常駐とライフサイクル

- OS サインイン後の自動起動を利用者が有効・無効にできる。
- 起動時にランチャーや設定画面を表示せず、通知領域常駐または不可視ウィンドウで待機する。
- アプリは原則 1 インスタンスとし、二重起動時は既存プロセスへ要求を転送する。
- 通知領域メニューから「ランチャーを表示」「設定」「終了」を実行できることを推奨する。
- 常駐中に異常終了した場合、次回起動時に診断可能なログを残す。

#### ランチャーの表示・非表示

- Windows キーによってスタートメニューが開いたタイミングに合わせてランチャーを表示する。
- Windows キー連動が利用できない環境に備え、利用者が任意の代替グローバルショートカットを登録、変更、無効化できる。
- スタートメニューが表示されたモニターにランチャーを配置する。
- Windows 11 のタスクバーアイコン配置は中央揃え／左揃えの両方に対応する。
- タスクバーが画面下端にある場合のみ動作する。それ以外の配置を検出した場合は監視・ランチャー機能を開始せず、対応外であることをエラー表示する。
- ランチャーはスタートメニューの右側に固定する。
- スマートフォン連携パネルが表示されている場合は、そのさらに右側に配置する。
- ランチャーの基準幅は、WQHD・表示スケール 125% の環境で約 400 effective pixel（表示スケール適用後の論理ピクセル）を目安とする。
- 右側に必要な表示領域を確保できない場合はランチャーを表示せず、領域不足エラーを表示する。縮小、列数削減、スクロールによる横幅補償、他 UI への重なりは行わない。
- 次の場合にランチャーを閉じる。
  - アクションが正常に起動され、その項目の実行後動作が「成功後に閉じる」のとき
  - ランチャー外がクリックされたとき
  - スタートメニューが閉じたと判断できたとき
  - Escape キーが押されたとき
  - セッションロック、ディスプレイ構成変更などで安全な表示を維持できないとき
- 短時間の連続入力、キーリピート、フォーカス移動で点滅や多重表示を起こさない。
- ランチャー表示中もスタートメニューの検索、キーボード操作、フォーカス挙動を維持し、可能な限りスタートメニューと一体の UI のように振る舞う。
- スタート連動表示では、スタートメニューと同じタスクバー上端を起点に、ランチャー全体を下から上へ進入させる。非表示時は逆方向へ退出させる。
- スタートメニューとランチャーは別プロセスであり、同一の Visual やアニメーション進行率を共有できないため、一体感は矩形、下端基準、方向、開始時刻、時間、イージングを揃えることで実現する。
- スタート連動の初期表示は非アクティブとし、スタートメニューからフォーカスを奪わない。ランチャーがポインターまたはキーボード操作を得た後は対話状態へ移行し、スタートが閉じても表示を維持する。
- 表示・非表示は明示的な状態機械で管理し、進入中の終了、退出中の再表示、キー連打を現在の表示状態から安全に反転できるようにする。

> 重要: Windows ロゴキー単体を `RegisterHotKey` で登録することはできず、Windows キーを含むショートカットは OS 用に予約されている。したがって「Windows キーを横取りする」のではなく、低レベル入力監視とスタート UI の出現確認を組み合わせ、OS 本来のスタート動作を妨げない方式を技術検証する。公開 API だけでスタートメニューの開閉・位置・スマートフォン連携パネルを確実に取得できる保証はない。

#### アクションとコントロール

- 初回リリースで次の項目種別を扱う。
  - ボタン: アプリ、ファイル、フォルダー、URL、コマンド、Windows URI の起動
  - 循環切り替え: 登録済み音声出力デバイス、または複数コマンドの順送り実行
  - スライダー: 現在の既定音声出力デバイスのマスター音量
- 項目は共通して ID、種類、表示名、アイコン、色、配置、実行後の表示方針、有効状態を持つ。
- アクション実装はプラグイン可能な内部インターフェースとし、将来のウィジェットや追加機能から UI を分離する。
- 実行中は二重実行を防止でき、必要に応じて進行中表示を出す。

#### 音声出力先の循環切り替え

- 1 つの切り替えコントロールに、音声出力デバイス ID の順序付きリストを保持する。保存可能な最小登録数は 2 台とし、固定の最大登録数は設けない。
- 操作時には OS の現在の既定出力デバイスを取得し、登録リスト内の現在位置より後ろにある最初の利用可能なデバイスへ切り替える。末尾まで候補がなければ先頭から探す。
- 現在の既定出力デバイスが登録リストにない場合は、登録順で最初の利用可能なデバイスへ切り替える。現在位置を設定ファイルやメモリ上のカウンターだけで管理しない。
- OS 側で既定出力デバイスが変更された場合も、次回操作時は変更後の既定デバイスを基準に循環を続ける。
- ランチャー上には現在の既定出力デバイス名を表示し、切り替え成功後に表示を更新する。切り替え要求に失敗した場合は現在表示を維持してエラーを示す。
- 設定画面では登録デバイスの追加、削除、並べ替えを提供し、同じデバイス ID の重複登録を禁止する。

#### コマンドの循環実行

- 1 つの循環切り替えコントロールに、表示名と実行定義を持つコマンドステップを 2 件以上、順序付きで登録できる。
- 操作するたびに現在の実行位置にあるコマンドを 1 件だけ起動する。末尾の次は先頭へ戻る。
- 任意コマンドの外部状態は共通方式では判定できないため、実行位置はアプリの稼働中だけ保持する。アプリ再起動、設定変更またはステップ並べ替え後は先頭から再開する。
- ランチャー上には現在状態ではなく `次: {ステップ表示名}` を表示し、利用者が次に実行される内容を操作前に確認できるようにする。
- 各ステップには既存のコマンド実行定義を再利用し、実行対象、引数、作業ディレクトリ、コマンド画面を表示しない設定を持たせる。
- 例として `拡張: DisplaySwitch.exe /extend` と `PC 画面のみ: DisplaySwitch.exe /internal` を登録した場合、操作順は `拡張 → PC 画面のみ → 拡張` となる。
- 「失敗時は同じ操作を再実行する」を項目ごとに設定でき、初期値はオンとする。オンの場合は起動要求失敗時に実行位置を維持し、オフの場合は失敗しても次へ進める。どちらもエラーは表示する。起動後のプロセス終了、終了コードおよび実際のシステム状態は追跡しない。

#### コマンド実行

- `cmd.exe`、PowerShell、任意実行ファイルをウィンドウ非表示で起動できる。
- `.bat` ファイルをダブルクリックした場合に近い「起動して引き渡す」動作とする。起動成功後、アプリは終了待機、出力取得、監視、キャンセル、終了コード判定を行わない。
- 実行ファイル、引数、作業ディレクトリを設定できる。初回リリースでは環境変数、タイムアウト、終了コード判定は扱わない。
- 引数は文字列連結ではなく、可能な限り `ProcessStartInfo.ArgumentList` 相当で分離して扱う。
- 標準出力・標準エラーは取得しない。
- 管理者権限が必要なコマンドは通常実行と明確に区別し、実行時に UAC が表示され得ることを示す。昇格実行と完全な非表示実行の両立は保証しない。
- シェル文字列を扱う項目は設定画面で危険性を表示し、インポートした設定を無条件に実行しない。

#### 設定画面

- ランチャー項目の追加、編集、削除、並べ替え、有効・無効切り替えを行える。並べ替え後も対象項目の選択と表示位置を維持する。
- 項目種別に応じた入力欄と入力検証を提供する。
- ボタンと循環切り替えは、項目ごとに「成功後に閉じる」または「表示したまま」を選択できる。既定値は「成功後に閉じる」とする。
- 設定画面上部の一般設定は横並びを維持し、編集領域は薄い境界で区切る。ウィンドウは表示先モニターのDPIと作業領域に応じて収まる大きさへ調整する。
- グリッドの列数、表示サイズ、表示側、テーマ、アニメーション、自動起動等を設定できる。
- ランチャーは単一ページとし、全項目を縦方向の単一スクロール領域に表示する。項目フォルダー機能は持たない。
- レイアウトは「標準」と「コンパクト」を選択できる。標準は現在のカード幅を維持して最大2列、コンパクトは通常ボタンを最大4列とし、循環切り替えとスライダーは通常ボタン2列分の幅を使う。
- ランチャー幅は固定物理ピクセルではなくモニターDPIから算出し、スタートメニュー右側の空き幅が不足する場合は標準を2列から1列、コンパクトを4列から3列、2列へ減らす。
- マウスホイールでページ間を含めて滑らかに縦スクロールする。ランチャーを表示するたびスクロール位置を最上部へ戻す。
- 代替グローバルショートカットの登録、変更、無効化、競合検証を行える。初期ショートカットは登録しない。競合時は登録を受け付けず、利用者に別の組み合わせを再登録させる。
- 音声出力先切り替え用に 2 台以上のデバイスを順序付きで登録できる。切り替え操作のたびに登録順で次の利用可能なデバイスへ進み、末尾から先頭へ循環する。
- 切断、無効化または利用不能な登録デバイスは切り替え時に飛ばす。現在の既定デバイス以外に切り替え可能な登録デバイスがない場合のみ、切り替えコントロールをグレーアウトして操作不能にする。
- 循環切り替え項目は「音声出力」と「コマンド」のモードを選択できる。コマンドモードでは 2 件以上のステップを追加、編集、削除、並べ替えできる。
- 音量スライダーは、現在の既定出力デバイスのマスター音量を操作する。アプリ別音量は対象外とする。
- 変更は「適用」または自動保存のどちらかに統一する。初期案は、項目編集を明示保存、軽微な外観設定を即時反映とする。
- 設定のエクスポート・インポートは初回リリース候補とし、機密値を平文で含めない。
- 設定破損時にバックアップから復旧または初期化できる。

### 1.3 非機能要件

- UI スレッドを入力監視、プロセス実行、ファイル I/O、音声デバイス列挙、同期COM待機でブロックしない。
- ランチャー表示のホットパスでは、同期I/O、音声デバイス列挙、`UpdateLayout`、同期 `RedrawWindow`、詳細診断を実行しない。
- 音声デバイス一覧、既定デバイス、マスター音量はキャッシュした状態を即時表示し、デバイス変更イベントまたはバックグラウンド処理で更新する。定期的な音声ポーリングは行わない。
- アイドル時 CPU 使用率は通常ほぼ 0% を目標とする。UI Automationはイベント駆動を基本とし、無関係なフォーカス変更ではデスクトップツリーを走査しない。
- Windowsキー後の高速ポーリングはイベント欠落を補う短時間フォールバックに限定し、ランチャー非表示かつスタート待機中でない場合は停止する。
- UI Automationイベント登録には時間がかかる場合があるため、監視の準備状態を管理し、登録完了前でも限定的な直接スキャンまたは代替操作が使えるようにする。
- 2026-07-20の実測ではUI Automationイベント登録に約2.4秒かかったため、コールドスタートの監視準備時間は表示ホットパスと分けて計測・改善する。
- アイドル時のワーキングセットは 500 MB 以下を必須とし、可能な限り小さくする。
- スタート表示確認からランチャー表示要求までは50ms以内を目標とする。現行実測の184～391msをベースラインとし、キャッシュ済み状態では10～50msへ短縮する。
- Windowsキーから最初のランチャー描画までは、監視準備済みの通常状態で100～150msを初期目標とする。コールドスタートはUI Automation準備時間を分離して計測する。
- 表示開始とアニメーション完了の体感速度は Windows 11 25H2 のスタートメニューと同等を目標とし、60Hz、120Hz、144Hzで評価する。
- 進入・退出はWindows Compositionで実行し、UIスレッドタイマーとフレームごとのHWND移動を使用しない。
- 進入は250msと `cubic-bezier(0, 0, 0, 1)`、退出は125msと `cubic-bezier(1, 0, 1, 1)` とする。退出は初期値167msから実機フィードバックにより短縮し、Opacityは83msで完了する。
- アニメーション中にUIスレッドへ16msを超える同期処理を置かず、位置飛び、終端の段付き、逆戻りを発生させない。
- Windows 11 のライト／ダークテーマ、アクセント色、角丸、影、Mica/Acrylic、モーション設定に追従する。
- ランチャーは一時的なlight-dismiss UIとしてDesktop Acrylicを第一候補とし、透明効果無効、バッテリー節約、RDP、ハイコントラストでは読みやすい単色へフォールバックする。
- 「アニメーションを表示しない」アクセシビリティ設定、DPI 100～300%、キーボード操作、スクリーンリーダー、高コントラストを考慮する。
- ローカル設定を基本とし、明示的な機能追加なしにネットワーク送信しない。
- 個人利用向けのローカル配布とし、自動更新機能は実装しない。パッケージサイズに上限は設けないが、UI と常駐処理は軽量化する。
- UI とメッセージは日本語のみとする。
- 対応外のタスクバー配置を検出した場合はエラーを表示し、その後アプリ全体を終了する。

### 1.4 初回リリース範囲

含む: 常駐、Windowsキー／スタート同期、登録可能な代替ホットキー、縦スクロール可能な単一ページのランチャー、複数モニター配置、スマートフォン連携パネル回避、ボタンアクション、起動後に監視しないコマンド実行、設定編集、ログ、音量調整スライダー、登録済みの複数音声出力デバイスまたはコマンドを順番に切り替えるコントロール。
含まない: 温度等のウィジェット、モニター輝度調整、クラウド同期、外部プラグイン配布基盤、管理者サービス。

モニター輝度調整は初回リリース完成後の追加機能とする。初回リリースの完成条件、性能測定および安定性試験には含めず、完成後に改めて対応機器と安全性を技術検証してから実装可否と範囲を確定する。

## 2. 画面構成

### 2.1 ランチャー画面 (`LauncherWindow`)

- 枠なし、タイトルバーなし、タスクバー非表示、常に最前面を基本とする非アクティブ型ウィンドウ。
- スタートメニュー横にカード状コンテナを表示する。
- ウィンドウ全体を毎フレーム移動せず、透明な `MotionHost` 内の `LauncherSurface` を単一のComposition Visualとして動かす。
- `MotionHost` は最終表示位置から対象モニターの作業領域下端までを移動路として持ち、範囲外のSurfaceをクリップする。
- 背景素材はWindow全体ではなく `LauncherSurface` の範囲に適用し、表示開始前に背景だけが四角く出ることを防ぐ。Windows App SDK 1.8ではテーマの `AcrylicInAppFillColorDefaultBrush` を使い、2.0以降への更新時に `SystemBackdropElement` と `DesktopAcrylicBackdrop` を再評価する。
- 構成:
  - ヘッダー（任意）: プロファイル名、設定ボタン
  - コンテンツ: 可変列グリッド
  - 項目: ボタン、トグル、スライダー、将来のウィジェット
  - 状態表示: 実行中、成功、失敗、無効
- アニメーション:
  - スタート連動表示はタスクバー上端から上方向へ250msで進入する
  - 非表示は下方向への移動と83msのフェードを組み合わせ、125msで退出する
  - 面全体を一つのVisualとして動かし、カード単位の位置アニメーションを同時に実行しない
  - バウンド、スプリング、オーバーシュート、回転、拡大縮小は使用しない
  - OS のモーション無効設定時はアニメーションを行わず、最終状態へ即時遷移する
- フォーカス方針は検証事項。表示時にスタートメニューからフォーカスを奪うとスタートが閉じる可能性があるため、初期表示は非アクティブ、キーボード操作開始時のみアクティブ化する案を優先する。

### 2.2 設定画面 (`SettingsWindow`)

- 通常の独立ウィンドウとして表示し、ランチャーとは別ライフサイクルで管理する。
- NavigationView を想定した構成:
  - 一般: 自動起動、代替グローバルショートカット、起動動作
  - レイアウト: 列数、サイズ、ページ、モニター別設定
  - 項目: 一覧、追加、削除、上下ボタンによる並べ替え、編集
  - 外観: テーマ、背景素材、アクセント、アニメーション
  - 詳細: ログ、設定フォルダー、復旧、バージョン
- 項目編集は右ペインまたはダイアログで、種別選択後に対応フォームを表示する。
- 上部設定は横並びとし、一覧と編集ペインを境界付きの領域として示す。表示先モニターの作業領域を超えない範囲でDPIに応じてウィンドウを拡大・中央配置する。
- 未保存変更がある状態で閉じる場合は破棄確認を行う。

### 2.3 通知領域メニュー

- ランチャーを表示
- 設定
- 一時停止／再開
- 終了

WinUI 3 単体では通知領域機能が限定されるため、Win32 相互運用または保守されているライブラリの採否を検証する。

### 2.4 エラー通知

- 操作失敗はランチャー内の短い通知または設定画面の InfoBar で示す。
- バックグラウンド障害は必要な場合のみ Windows 通知または通知領域で示す。
- コマンドの生ログや機密情報を一般通知に表示しない。

│  ├─ LauncherItemDefinition
│  ├─ ButtonActionDefinition
│  ├─ CycleActionDefinition
│  ├─ CommandCycleStepDefinition
│  ├─ SliderActionDefinition
│  ├─ WindowPlacement
│  └─ ExecutionResult
├─ Services
│  ├─ ILauncherCoordinator            # 入力・Start・Focus・Motion状態を統括
│  ├─ ILauncherMotionService          # Composition進入・退出・反転・完了通知
│  ├─ IStartMenuObserver              # スタート出現推定、監視準備状態
│  ├─ IGlobalInputMonitor             # Windows キー等の監視

   └─ IWidgetProvider                 # 将来用、初回は実装しない

主な責務:

- `LauncherCoordinator`: 入力、スタート状態、ウィンドウ状態を状態機械として統合し、重複表示や競合を防ぐ。
- `LauncherMotionService`: `LauncherSurface` のTranslationとOpacityをCompositionで実行する。完了は `CompositionScopedBatch` で通知し、世代IDによりキャンセル済み完了通知を無視する。
- `StartMenuObserver`: 公開 API で不足する部分を WinEvent/UI Automation 等で観測する。監視の準備状態、イベント中心の観測、短時間フォールバックを管理し、OS 依存コードをここだけに閉じ込める。
- `WindowPlacementService`: 対象モニターの作業領域、DPI、タスクバー、推定スタート矩形、回避矩形から最適位置を純粋計算する。
- `ActionExecutor`: 定義に対応するハンドラーを選び、キャンセル、二重実行防止、結果通知を共通化する。
- `AudioStateService`: デバイス一覧、既定出力、マスター音量をキャッシュし、OSのデバイス変更通知で更新する。表示要求から同期列挙を呼び出さない。
- `SettingsRepository`: スキーマバージョン付き JSON、原子的保存、バックアップ、移行、検証を担う。

## 4. データの流れ

### 4.1 起動

```text
OS/利用者による起動
  → 単一インスタンス判定
  → DI・ログ・設定読み込み
  → 設定検証／必要なら移行・復旧
  → 音声状態は未取得または前回キャッシュでViewModelを構築
  → 入力監視を先に開始
  → UI Automationワーカーを開始し、監視準備状態を公開
  → ウィンドウとLauncherSurfaceを生成・初回レイアウトしても表示しない
  → 音声状態をバックグラウンド取得し、以後は変更イベントで更新
  → 待機
```

起動処理はランチャー利用可能状態と補助情報の準備を分離する。UI Automationイベント登録や音声デバイス列挙が遅れても、入力監視、代替ホットキー、設定画面、終了操作を不必要に待たせない。

### 4.2 ランチャー表示

```text
Windowsキー候補イベント
  → debounce
  → LauncherCoordinator: AwaitingStartConfirmation
  → StartMenuObserver がイベントまたは短時間フォールバックでスタート表示を確認
  → 表示中のスタート矩形または対象モニターを推定
  → DisplayService が作業領域・DPIを取得
  → SystemPanelObserver が回避領域を取得（取得不能なら安全な既定値）
  → WindowPlacementService がスタート／スマートフォン連携パネル右側の位置を計算
  → MotionHostを最終位置へ一度だけ配置
  → LauncherViewModel はキャッシュ済み状態を即時公開
  → スクロール位置を最上部へ戻す
  → LauncherWindow を非アクティブ表示
  → LauncherMotionService がLauncherSurfaceをタスクバー上端からComposition進入
  → Composition完了通知
  → VisibleWithStart または VisibleInteractive
```

表示確認からComposition開始までの経路では、音声列挙、ファイルI/O、同期COM待機、強制レイアウト、同期再描画、詳細診断を実行しない。スタート未確認または配置失敗の場合はSurfaceを表示しない。

### 4.3 ランチャー非表示

```text
Start hidden / Escape / outside click / action success
  → LauncherCoordinator が対話状態と終了理由を評価
  → VisibleInteractive ならStart終了だけでは維持
  → RequestExit(reason)
  → LauncherMotionService が下方向Translation＋Opacityで退出
  → Composition完了通知
  → Windowを非表示
  → Hidden、フォールバック監視停止
```

退出中に再表示要求が来た場合は、現在のComposition表示値から進入へ反転する。古い完了通知が現在状態を上書きしないよう、アニメーション世代を識別する。

### 4.4 アクション実行

```text
View の操作
  → ICommand / ViewModel
  → IActionExecutor
  → 種別別 IActionHandler
  → OS、Process、Shell 等へ起動を要求
  → 起動要求の成否のみを ExecutionResult として返す
  → 成否表示・ログ（起動後のプロセスは追跡しない）
  → ポリシーに従いランチャーを閉じる／維持
```

### 4.5 設定変更

```text
設定フォーム
  → ViewModel の入力検証
  → 編集用コピーを更新
  → 保存要求
  → ドメイン検証
  → SettingsRepository が一時ファイルへ書き込み
  → 置換＋バックアップ
  → 設定変更イベント
  → LauncherViewModel が再構築
```

設定ファイルはスキーマバージョンを持つ JSON を第一候補とする。書き込み途中の破損を避けるため、一時ファイルへの保存後に置換する。UI にバインドするオブジェクトと永続化モデルは分離し、キャンセル可能な編集を実現する。

## 5. エラー処理方針

### 5.1 原則

- 例外を握りつぶさず、境界ごとに利用者向け結果と診断ログへ変換する。
- 回復可能な項目単位の失敗でアプリ全体を終了しない。
- UI スレッド未処理例外、非同期未観測例外、プロセス終了直前の障害を集約して記録する。ただし未知の致命的状態で無理に継続しない。
- ログには時刻、重大度、イベント ID、操作 ID、例外種別を含め、コマンド引数、ユーザー名、パス等は必要に応じてマスクする。

### 5.2 ケース別

| ケース | 利用者への動作 | 内部処理 |
|---|---|---|
| 設定ファイルなし | 初期設定で開始 | 新規作成 |
| 設定 JSON 破損 | 復旧した旨を表示 | バックアップ→既定値の順に復旧、破損ファイル保存 |
| スキーマ不一致 | 移行結果を通知 | 段階的マイグレーション、失敗時ロールバック |
| アプリ／ファイルなし | 項目上にエラー | 実行せずパスを記録（表示時は配慮） |
| コマンドの起動要求失敗 | 失敗表示 | Win32／プロセス起動例外を記録 |
| 循環コマンドの起動要求失敗 | 失敗表示し、設定に従って同じステップを再試行または次へ進む | 失敗したステップ名と例外を記録 |
| 登録音声デバイスの切断・無効化 | 対象デバイスを飛ばして次の利用可能なデバイスへ切り替える。切り替え先がない場合のみコントロールをグレーアウト | デバイス再接続イベントを待ち、利用可能性と循環順を再評価 |
| グローバルショートカット競合 | 登録せず再登録を要求 | 既存設定を維持し、競合内容を表示 |
| 権限不足 | UAC または説明 | 自動で無制限昇格しない |
| 入力フック失敗 | 代替ホットキーを案内 | 再登録を限定回数試行 |
| スタート検出不能 | ランチャーを出さない／代替位置 | 検出方式をフォールバック |
| UI Automationイベント登録が遅い | 監視準備中を内部状態として扱い、代替ホットキーは利用可能にする | 登録をバックグラウンド継続し、Windowsキー後は直接スキャンの可否を判定 |
| UI Automationの無関係なフォーカス変更 | 利用者への表示なし | 待機状態と対象プロセスを確認し、不要なツリー走査と詳細ログを行わない |
| 音声状態キャッシュ未準備 | 前回値または取得中表示でランチャーを先に表示 | バックグラウンド取得し、完了後にViewModelへ通知 |
| Composition進入・退出の割り込み | 現在位置から自然に反転 | 世代IDで古い完了通知を破棄し、最終状態を一度だけ確定 |
| Acrylic利用不能 | 読みやすい単色で表示 | OSのフォールバックを使用し、コントラストを検証 |
| モニター切断・DPI変更 | 一旦非表示して再配置 | キャッシュ破棄、再計算 |
| 右側の表示領域不足 | ランチャーを表示せずエラー | 必要幅と利用可能幅を記録 |
| タスクバーが下端以外 | 対応外エラー後にアプリ終了 | 監視・ランチャー機能を開始しない |
| UI Automation 対象消失 | 無視して再観測 | COM 例外を境界で処理 |

7. 単一ページ内の項目を、マウスホイールで滑らかに縦スクロールできるようにする。
8. ランチャー表示ごとにスクロール位置を最上部へ戻す。

### Phase 4: 設定画面（主要実装完了・細部調整保留）

1. 単一の設定画面でランチャー項目を編集できるようにする。
2. 項目の追加、編集、削除、並べ替え、入力検証を実装する。
3. ショートカットの実行対象と表示内容を登録できるようにする。引数、作業ディレクトリ、コマンド画面の表示は補助的な詳細設定として扱う。
4. 音声出力デバイスを 2 台以上登録し、追加、削除、並べ替え、再登録できるようにする。登録数に固定上限は設けない。
5. 循環切り替え項目で「音声出力」または「コマンド」を選択できるようにし、コマンドステップの追加、編集、削除、並べ替えを実装する。
6. コマンドステップごとに表示名、実行対象、引数、作業ディレクトリ、コマンド画面の表示方針を設定できるようにする。
7. スマートフォン連携パネルの有無、標準／コンパクトの適応レイアウト、外観、自動起動を設定できるようにする。
8. 設定保存失敗や破損をユーザーへ通知し、安全な既定値へ復旧できるようにする。
9. 既存の音声出力切り替え設定を、音声出力モードの循環切り替え定義へデバイス順を維持したまま移行する。

実装状況: 1～6、8、9と、7のスマートフォン連携、標準／コンパクト、自動起動は実装済み。さらに項目ごとの実行後動作、上下移動後の選択・スクロール追従、DPI対応の設定画面サイズ、診断画面を追加した。テーマ等の外観設定、代替グローバルショートカットの登録UI、細かな余白・文言・装飾は未完了または採用判断待ちであり、Phase 5の機能試験を妨げないものは後回しとする。

### Phase 4.5: モーション・表示ホットパス再設計（コード置換完了・実機検証継続）

## 8. 不明点とリスク

### 8.1 確定した要件

- Windows 11 25H2 以降、x64、日本語のみを対象とする。
- Windows キー連動に加えて、利用者が代替グローバルショートカットを登録できる。
- 下端タスクバーの中央揃え／左揃えに対応し、それ以外のタスクバー配置では起動不可エラーとする。
- ランチャーはスタートメニューの右、スマートフォン連携パネル表示時はさらにその右へ配置する。
- WQHD・125% で約 400 px の幅を目安とし、右側の領域不足時は表示を中止してエラーを表示する。
- スタートメニューの検索・キーボード操作を維持する。
- 登録した 2 つ以上の音声出力デバイス、または 2 件以上のコマンドを登録順に循環する切り替えコントロールと、音量スライダーを提供する。
- コマンドは起動後に終了待機、出力取得、監視、操作を行わない。
- 設定とランチャーページはそれぞれ 1 つとし、項目フォルダーは持たない。
- 全ページを縦に連結して滑らかにマウスホイールスクロールし、ランチャー表示のたび最上部へ戻す。
- 利用不能な登録音声デバイスは循環対象から一時的に除外し、現在の既定出力以外に切り替え先がない場合のみ切り替えコントロールをグレーアウトして操作不能にする。
- 音量スライダーは現在の既定出力デバイスのマスター音量を操作し、アプリ別音量は扱わない。
- 代替グローバルショートカットに初期値は設けず、競合時は別の組み合わせを再登録させる。
- `.bat`、`.cmd`、`.exe`、PowerShell スクリプトは、起動後に追跡しない同一方針で扱う。
- 対応外タスクバー配置では、エラー表示後にアプリ全体を終了する。
- 個人利用向けで、自動更新は行わない。
- アイドル時メモリは 500 MB 以下、表示体感はスタートメニュー同等を目標とする。
- スタート連動モーションはタスクバー上端を起点とし、Compositionでランチャー面全体を移動する。
- ランチャーはtransient UIとしてSurface単位のAcrylicを使い、非対応環境では単色へフォールバックする。Desktop Acrylicは対応APIへ更新した時点で再評価する。
- 表示ホットパスでは同期音声列挙、強制レイアウト、同期再描画を行わない。

### 8.2 技術検証後に決定する事項

1. スタートメニューとのフォーカス共存は必須条件とせず、技術検証で実現できた範囲と副作用を確認して最終決定する。
2. スタートメニューおよびスマートフォン連携パネルの位置・表示状態を Windows 11 25H2 以降で安定して取得できるかを確認し、利用可能な検出方式とフォールバックを確定する。
3. 250ms進入と125ms退出を初期値として、スタートメニューとの知覚上の同期を実測し、必要なら時間とイージングを再調整する。
4. 透明MotionHostとSurface単位のテーマ背景で、Surface外の透明領域が入力を遮らず、角丸・影・Acrylicを一体で動かせるかを実機確認する。Windows App SDK 2.0以降への移行時は `SystemBackdropElement` でも再検証する。

### 8.3 主要リスク

| リスク | 影響 | 対策 |
|---|---|---|
| Windows キー単体は通常のグローバルホットキーとして登録不可 | 中核 UX が実現不能または不安定 | OS 動作を妨げない観測方式を先行検証し、代替ホットキーを用意 |
| スタートメニューに安定した公開連携 API がない | OS 更新で検出・位置取得が壊れる | アダプター隔離、複数方式、機能フラグ、OS ビルド試験 |
| スマートフォン連携パネルの公開矩形 API がない可能性 | 重なりを完全には防げない | UI Automation／ウィンドウ矩形を検証し、取得不能時は利用者指定の予約領域を使う。必要幅を確保できなければ表示しない |
| 非アクティブ表示と入力操作が競合 | スタートが閉じる、ランチャーを操作できない | フォーカス状態機械とプロトタイプで早期検証 |
| 低レベルキーボードフックのセキュリティ製品との相性 | 警告、入力遅延、利用不可 | 処理最小化、キーを遮断しない、署名、無効化可能、代替操作 |
| WinUI 3 と Win32 相互運用の複雑さ | ハンドルリーク、クラッシュ | Infrastructure 隔離、確実な解除、長時間試験 |
| Mica/Acrylic と常駐ウィンドウのコスト | メモリ増、低性能 GPU で遅延 | 効果を設定可能にし、低負荷フォールバックを用意 |
| UIスレッドタイマーによるHWND移動 | フレーム欠落、整数座標の段付き、リフレッシュレート非同期 | Composition Visualへ移行し、HWND配置は表示前の一回だけにする |
| 表示直前の音声デバイス同期列挙 | 反応が184～391ms遅れ、期限切れDispatcher処理がアニメーション開始後に集中 | 音声状態キャッシュ、変更イベント、バックグラウンド更新 |
| UI Automationイベント登録の遅延 | 起動直後のスタート検出が遅れる | 準備状態を公開し、登録と直接スキャンを分離、代替ホットキーを先に利用可能にする |
| グローバルフォーカスイベントごとのツリー走査 | 操作中の常駐CPU・ログI/O増加 | 状態と対象プロセスでゲートし、スタート待機外では詳細走査しない |
| 透明MotionHostが入力を遮る | 見えない領域で他アプリを操作できない | Host範囲最小化、ヒットテスト制御、退出完了後のWindow非表示を試験 |
| 起動後に追跡しないコマンド実行 | コマンド注入、秘密漏えい、子プロセスの異常を検出不能 | 引数分離、入力検証、ログ秘匿、起動後は責任範囲外であることを明示 |
| Release trimming と WinUI／リフレクション | Release のみ動作不良 | trimming 方針を早期検証し、必要な型を保持または無効化 |
| 温度取得に標準 API がない／機種差が大きい | 将来ウィジェットが不安定 | `IWidgetProvider` で隔離し、対応センサーと外部依存を別途選定 |

## 9. 初期アーキテクチャ判断

- UI は WinUI 3、MVVM は `CommunityToolkit.Mvvm` 採用候補、DI／ログは `Microsoft.Extensions.*` 採用候補とする。依存バージョンは実装開始時に確定する。
- `AppWindow` と HWND 相互運用を併用し、OS 固有機能を View／ViewModel から隔離する。
- 表示制御は単純なイベント直結やトグルではなく、`LauncherCoordinator` の状態機械で行う。
- HWNDは配置と表示・非表示の境界として扱い、フレームごとのモーションには使わない。進入・退出は `LauncherSurface` のComposition TranslationとOpacityで実行する。
- モーション完了はUIタイマーではなくCompositionの完了通知で扱い、割り込みは現在の表示値から反転する。
- 音声状態は `AudioStateService` がキャッシュし、ViewModel構築時とランチャー表示時に同期列挙しない。
- UI Automationは専用STAスレッドへ隔離し、監視準備状態、対象状態によるゲート、イベント欠落時だけの有界フォールバックを持つ。
- 一時的なランチャー面にはDesktop Acrylicを優先し、Window全体へMicaを常時適用する構成は再評価する。
- アクションは定義データと実行ハンドラーを分離する。外部 DLL を読み込む本格的プラグイン機構は、セキュリティと互換性が固まるまで導入しない。
- 設定はローカル JSON を基本とし、スキーマ移行とバックアップを初期から設ける。
- 初回の開発ゲートは「スタートメニューとの共存を公開／許容 API の範囲で安定して実現できるか」の技術スパイクとする。
