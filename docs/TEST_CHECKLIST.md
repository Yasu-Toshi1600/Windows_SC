# Windows_SC 簡易テストチェックリスト

更新日: 2026-07-22  
対象: 一人で開発・配布確認を行う場合の標準手順

## 1. 基本方針

すべての環境と異常系を毎回試す必要はない。変更の大きさに応じて次の3段階を使う。

| 段階 | 実施時期 | 所要時間の目安 |
|---|---|---:|
| A: 日常確認 | 普段の修正後 | 5～10分 |
| B: 配布前確認 | 知人へ渡す前、バージョン更新時 | 30～45分 |
| C: 追加確認 | 関係する機能を大きく変更したとき、不具合を再現するときだけ | 必要な分だけ |

通常はAだけでよい。知人へ新しいZIPを渡す前にAとBを行う。詳細な全項目は[Phase 5～6 詳細テスト手順書](PHASE5_6_TEST_PROCEDURE.md)を参照するが、個人配布のたびに全項目を実施する必要はない。

## 2. 判定方法

- `PASS`: 期待どおり動作した。
- `FAIL`: 再現可能な不具合があった。
- `N/A`: その環境では試せない。例として、モニター1台の環境における複数モニターテストが該当する。

必須項目にFAILがなければ次へ進める。N/Aは不合格ではない。別PCで確認できたときに結果を補う。

不具合が出た場合だけ、次を残す。

```text
アプリバージョン:
操作:
期待した動作:
実際の動作:
再現回数:
発生時刻:
通常ログ:
詳細ログ: 必要な場合のみ
```

## 3. A: 日常確認

コードまたは設定画面を変更した後に行う。

### A-1 ビルドと起動

- [ ] x64 DebugまたはReleaseがエラーなしでビルドできる。
- [ ] 起動時にランチャーや設定画面が勝手に開かず、トレイアイコンが1個表示される。
- [ ] 終了後にプロセスとトレイアイコンが残らない。

### A-2 表示と終了

- [ ] Windowsキーで5回開閉できる。
- [ ] スタートボタンのクリックで5回開閉できる。
- [ ] `Ctrl+Alt+Space`で表示と非表示を切り替えられる。
- [ ] `Ctrl+Alt+Space`で表示後、ランチャーへ一度も触れず画面外をクリックして閉じる。
- [ ] 通常の画面外クリックとEscapeですぐ閉じる。

### A-3 基本操作

- [ ] 登録済みボタンを1個実行でき、成功後に閉じる。
- [ ] 設定画面を開き、項目を1つ変更して「保存」するとランチャーへ反映される。
- [ ] 保存せず設定画面を閉じた変更はランチャーへ反映されない。

A-1～A-3がすべてPASSなら、通常の修正確認は完了とする。

## 4. B: 配布前確認

Aを実施した後、知人へ新しい版を渡す前に行う。

### B-1 ランチャーの入口

- [ ] Windowsキーを10回連続で開閉して、起動漏れ、二重表示、遅い退出がない。
- [ ] スタートボタンを通常速度で10回開閉できる。
- [ ] スタートボタンを2～3回やや速く操作しても、その後操作不能にならない。
- [ ] トレイの「ランチャーを表示」「設定」「終了」が動く。

連打速度を競うテストは不要である。通常利用より少し速い操作で状態が取り残されないことだけ確認する。

### B-2 項目と設定

- [ ] アプリ／ファイル起動を1件実行できる。
- [ ] コマンドまたはbatを1件実行できる。
- [ ] 2件以上の循環切り替えが`1 → 2 → 1`の順で動く。
- [ ] 音声デバイス循環を利用する環境では、切り替えと再接続後の復帰を確認する。
- [ ] 音量スライダーを安全な範囲で動かせる。
- [ ] 項目の追加、削除、上下移動後も対象を見失わない。
- [ ] ボタンには「実行後」が表示されず、循環切り替えには表示される。
- [ ] 再起動後も保存した内容、順序、レイアウトが維持される。

利用していない項目種類はN/Aでよい。ただし、新しく変更した種類は必ず確認する。

### B-3 レイアウトとモニター

- [ ] 標準／コンパクトを切り替えると列数が変わる。
- [ ] 文字、ボタン、設定項目が見切れず操作できる。
- [ ] 複数モニター環境では、スタートとランチャーが同じモニターへ表示される。
- [ ] モニターが1台だけなら複数モニター項目はN/Aとする。

すべてのDPI、解像度、リフレッシュレートを一人で揃える必要はない。手元と知人PCで異なる構成が確認できれば、その結果を利用する。

### B-4 ログと配布物

- [ ] x64 Releaseをエラーなしでビルドできる。
- [ ] トラブルシューティングの環境情報にアプリ版、OSビルド、モニター情報が表示される。
- [ ] データフォルダーを開き、配下の`Logs`で通常ログを確認できる。
- [ ] 通常ログに起動時の環境情報があり、モニター構成等を変えた時だけ新しい環境情報が追記される。
- [ ] 詳細ログを有効化・無効化できる。
- [ ] 配布ZIPのアプリ版が今回のバージョンと一致する。
- [ ] 可能ならVisual Studioと.NET SDKのない別PCで起動できる。

別PC確認を毎回依頼できない場合、コード変更だけの配布では前回のクリーン起動結果を流用してよい。ランタイム、配布方式、プロジェクト設定を変更した場合は再確認する。

## 5. 知人へ依頼する最小確認

知人には次の5項目だけ依頼すればよい。

1. 起動してトレイアイコンが出る。
2. Windowsキーとスタートボタンクリックで5回ずつ開閉できる。
3. スタートとランチャーが同じモニターへ表示される。
4. 標準／コンパクトで見た目が変わる。
5. 設定を1件保存し、再起動後も残る。

問題がなければログ提出は不要である。不具合が出た場合だけ、環境情報、発生時刻、通常ログを受け取り、必要な場合に限って詳細ログを有効にする。

## 6. C: 変更内容に応じた追加確認

次の項目は毎回実施しない。該当部分を変更した場合、または問題が疑われる場合だけ行う。

| 変更・状況 | 追加する確認 |
|---|---|
| スタート検出、Windowsキー、モーションを変更 | 途中反転20回、スタートクリック20回、Windowsキーショートカット、詳細ログ時間 |
| モニター配置、DPI、レイアウトを変更 | 異なるDPIの2画面、主画面変更、負座標、モニター再接続 |
| トレイ、常駐、終了を変更 | Explorer再起動、二重起動、終了後のプロセス残留 |
| 設定保存、保存場所を変更 | 不正JSON、書込み失敗、旧データ移行、バックアップ復旧 |
| 音声処理を変更 | 切断、再接続、既定デバイス変更、スリープ復帰 |
| ログ、匿名化を変更 | ユーザー名、実パス、UI Automation要素名、ログ削除 |
| 配布構成、.NET、Windows App SDKを変更 | クリーンPC、自己完結起動、自動起動、アンインストール |
| メモリ増加や停止が疑われる | 1～2時間の通常利用と、開始／終了時のメモリ・ハンドル比較 |

100回反転、8時間常駐、全DPI、全リフレッシュレート、RDP、スクリーンリーダーは、正式公開前または該当不具合の調査時だけ行う。知人向け初回配布の必須条件にはしない。

## 7. 配布可否

次を満たせば、知人向けの試験版を配布してよい。正式な安定版として公開する場合だけ、詳細テスト手順書のPhase 6条件も使用する。

- AがすべてPASS。
- Bで、変更した機能と基本起動にFAILがない。
- N/A項目の理由が分かっている。
- 既知の不具合が利用不能、データ破損、終了不能、意図しないコマンド実行につながらない。
- バージョンと配布物を区別できる。

細かな見た目、未所持の機器、特殊なDPI等の未確認は、試験版であることを伝えれば配布を止める理由にしない。

## 8. 簡易結果メモ

```text
日付:
バージョン:
コミット:
実施PC:

A: PASS / FAIL
B: PASS / FAIL / 一部N/A
C: 実施した項目だけ記載

不具合:
配布可否: 可 / 不可
```
