# 次に行う作業と判断が必要な内容

更新日: 2026-07-24
対象バージョン: `0.6.1-dev`

## 1. この文書の目的

Windows_SCの現在地点から、次に実施する作業と、実装または配布の前に方針を
決める必要がある内容を分けて管理する。

詳細な不具合履歴は
[修正課題・ドキュメント整理表（2026-07-23）](archive/REMAINING_WORK_AND_DOCUMENTATION_AUDIT_2026-07-23.md)、
日常の確認方法は[簡易テストチェックリスト](TEST_CHECKLIST.md)を正とする。
この文書では、すでに完了した修正をすべて再試験するのではなく、`0.6.1`へ
進むために必要な次の行動を扱う。

## 2. 現在地点

- 主要機能と優先度Aの不具合修正は、おおむね実装・確認済みである。
- ランチャーと設定画面のUI調整を`0.6.0-dev`で実装し、現在は`0.6.1-dev`として
  配布前確認を進めている。
- 標準／コンパクトの高さ、長い表示名、種類ラベル、音量項目、ホイール操作、
  設定一覧、トラブルシューティングのスクロール、保存後の復元は実画面で
  確認済みである。
- 起動対象のフォルダーアイコンから「ファイルを選択」と
  「フォルダーを選択」を選べるようにした。この変更はDebug／Release x64で
  警告・エラーなしでビルド済みだが、実画面確認とコミットはまだ行っていない。
- 現在の作業ツリーには、上記の選択機能、仕様書、テスト項目、変更履歴の
  未コミット差分がある。

## 3. 次に行う作業

### 3.1 ファイル／フォルダー選択の実画面確認

最初に今回の変更だけを確認する。

1. ボタン項目の実行種類を「アプリ・ファイル・URLを開く」にする。
2. フォルダーアイコンから「ファイルを選択」を選び、フルパスが入力されることを確認する。
3. 「フォルダーを選択」を選び、フォルダーのフルパスが入力されることを確認する。
4. それぞれの選択画面でキャンセルし、入力済みの内容が変わらないことを確認する。
5. 保存後にファイルとフォルダーを実行し、関連付けされたアプリまたは
   エクスプローラーで開けることを確認する。
6. 「コマンド／batを実行」ではフォルダーアイコンが表示されないことを確認する。

### 3.2 現在の差分を確定する

実画面確認で問題がなければ、次を一つのUI調整コミットとして保存して
`main`へpushする。

- v0.6のUI調整
- 実画面確認で見つかったUI修正
- ファイル／フォルダー選択
- UI仕様書、簡易テスト、変更履歴の更新

### 3.3 v0.6の最小回帰確認

全項目の総当たりは行わず、次だけを確認する。

- Windowsキー、スタートボタンクリック、`Ctrl+Alt+Space`、通知領域から表示できる。
- Windowsキー、スタート終了、画面外クリック、Escapeで閉じられる。
- 仮想デスクトップを切り替えた後も表示できる。
- 通常ボタン、コマンド、音声循環、コマンド循環、音量スライダーを各1回実行できる。
- 設定を保存し、再起動後も内容が残る。
- 通常ログへ新しい未処理例外が記録されていない。

スタート連打、複数モニター、詳細ログ等は、関連コードを変更していなければ
過去の確認結果を利用してよい。不具合が再発した場合だけ詳細試験へ戻る。

### 3.4 配布準備

配布方式を決めた後、次を行う。

1. 利用者向け`README.md`を配布内容に合わせて最終確認する。
2. `CHANGELOG.md`の未リリース内容を整理する。
3. 開発版を`0.6.1-beta.1`または`0.6.1-rc.1`へ更新する。
4. x64自己完結版を作成する。
5. 配布物の版と診断情報の版が一致することを確認する。
6. 手元以外のPCで簡易テストを行う。
7. 合格後に`0.6.1`へ更新し、同じソースから正式配布物を作成する。

## 4. 判断が必要な内容

| ID | 判断する内容 | 選択肢と影響 | 推奨 |
|---|---|---|---|
| D-01 | v0.6へ含める範囲 | UI調整までで区切るか、残る内部改善も追加するか。追加を続けるほど回帰確認の範囲が広がる。 | UI調整とファイル／フォルダー選択で機能を凍結し、以後は不具合修正だけにする。 |
| D-02 | 次の版番号 | `dev`を継続する、知人試験用の`beta.1`にする、正式候補の`rc.1`にする。 | 今回の実画面確認後を`0.6.1-beta.1`とし、別PC確認後を`rc.1`にする。 |
| D-03 | 初回配布方式 | ZIPの自己完結版、MSIX、両方を配布する。MSIXは導入と自動起動の検証項目が増える。 | 最初はZIPのx64自己完結版を基準にし、MSIXは必要性を確認してから追加する。 |
| D-04 | 自動起動の保証範囲 | 非パッケージ版だけ、MSIXだけ、両方を保証する。両方を保証する場合はR-04を両形式で試験する必要がある。 | 実際に配布する形式を必須試験とし、配布しない形式は正式版の合格条件から外す。 |
| D-05 | 対応するWindows | 現行どおりWindows 11 25H2・ビルド26200以降に限定するか、24H2へ広げるか。対象を広げる場合はスタート検出と対応環境検査の再試験が必要になる。 | 初回配布は現行範囲を維持し、24H2対応は実機またはVMを用意できた後に判断する。 |
| D-06 | R-10 起動処理の非同期化 | 音声初期化等をバックグラウンド化するか、現状を維持するか。変更は起動順序と終了処理へ影響する。 | v0.6では現状維持。起動遅延が再現した場合は先に区間ログを追加し、原因を絞ってから変更する。 |
| D-07 | R-11 非対応環境試験 | beta前に行うか、正式版前まで保留するか。安全な24H2またはx86 VMが必要になる。 | 知人向けbetaでは既知の未確認事項として扱い、正式版前に理由付き終了とプロセス残留なしを確認する。 |
| D-08 | R-12 実行中ログローテーション | 2MB超過時に即時ローテーションするか、現行の次回起動時処理を維持するか。 | v0.6は現行仕様を明記して維持し、長時間試験で実害が出た場合に実装する。 |
| D-09 | 更新機能 | v0.6へ自動更新または更新確認を追加するか。署名、配布元、失敗時復旧等の設計が必要になる。 | v0.6には含めず、配布手順が安定してから別フェーズで設計する。 |
| D-10 | 正式版の番号 | 身内向けの最初の安定版を`0.6.1`とするか、製品上の区切りとして`1.0.0`にするか。 | 当面は`0.6.1`で配布試験を行い、導入手順と主要環境の確認が揃った時点を`1.0.0`とする。 |

## 5. 現時点で判断し直す必要がない内容

次の内容は仕様確定・実装・確認済みのため、関連する不具合が出ない限り
再設計しない。

- 標準表示とコンパクト表示の基本レイアウト
- 長い表示名の最大2行表示と種類ラベル
- 音量スライダーの配置とホイール変化量
- 項目一覧とトラブルシューティングのスクロール
- 明示的な保存方式
- コマンド循環の「失敗時は同じ操作を再実行する」の廃止
- UI Automation要素名をログへ記録しない方針
- 詳細ログ期限を診断設定の適用時だけ更新する方針
- 仮想デスクトップ移動時の表示処理

## 6. v0.6の区切り

### betaへ進める条件

- ファイル／フォルダー選択の実画面確認が完了している。
- 最小回帰確認で重大な再発がない。
- 現在の差分がコミットされ、GitHubと同期されている。
- 配布方式と自動起動の保証範囲を決定している。

### 正式版へ進める条件

- 選択した配布形式でx64自己完結版を再現可能に作成できる。
- 手元と別PCで簡易テストに合格している。
- 選択した配布形式の自動起動を確認している。
- R-11の非対応環境試験を完了するか、正式版の対応環境表示と
  未確認リスクを明示している。
- README、変更履歴、バージョン、配布物の内容が一致している。

## 7. 推奨する直近の順序

現時点では、次の順序が最も小さい負担で`0.6.1`へ進められる。

1. ファイル／フォルダー選択を実画面確認する。
2. 現在の差分をコミットしてpushする。
3. D-03の配布方式とD-04の自動起動保証範囲を決める。
4. 最小回帰確認を行う。
5. READMEと配布手順を配布内容に合わせて確定する。
6. `0.6.1-beta.1`を作り、別PCで確認する。
7. betaで見つかった不具合だけを修正し、`rc.1`または正式版へ進む。
