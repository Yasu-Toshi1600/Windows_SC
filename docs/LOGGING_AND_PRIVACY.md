# ログ・詳細診断・プライバシー仕様

作成日: 2026-07-21
対象: Windows_SC Phase 5

## 1. 目的

通常利用で必要な障害情報を残しながら、UI Automation候補、座標、デバイス識別子、実行対象などの調査情報を通常ログから分離する。ログはローカル保存だけとし、Windows_SCからネットワークへ送信しない。

## 2. 保存場所

MSIX／非パッケージ実行のどちらでも、利用者から見える保存先を次へ統一する。

```text
%LOCALAPPDATA%\Windows_SC\
├─ Settings\
│  ├─ settings.json
│  └─ Backup\
│     └─ settings.corrupt-日時.json
└─ Logs\
   ├─ window-diagnostics.log
   ├─ window-diagnostics.previous.log
   ├─ window-diagnostics.detail.log
   └─ window-diagnostics.detail.previous.log
```

MSIXでは`%LOCALAPPDATA%\Windows_SC`だけをファイルシステム仮想化から除外する。設定画面には「データフォルダーを開く」だけを設け、そこから`Settings`と`Logs`へ移動できるようにする。データフォルダーが未生成または削除済みでも再作成してから開く。

旧版からの初回起動時は、ルート直下の`settings.json`、MSIXの`LocalCache\Local\Windows_SC`、`LocalState\Logs`を検出し、新構成へ自動移行する。複数の設定が見つかった場合は更新日時が最も新しいものを現行設定とし、残りは`Settings\Backup`へ退避する。

- UTF-8、BOMなし、各行にローカル日時とミリ秒を付ける。
- 通常ログファイルはロガー初期化時に作成し、起動直後でも保存場所を確認できるようにする。
- 通常ログと詳細ログはそれぞれ起動時に2MB以上ならpreviousへ移動する。
- previousは各種類1世代だけ保持する。
- 設定画面の削除操作では4ファイルを確認後に削除し、現在の環境情報を新しいログへ直ちに記録する。

## 3. 通常ログ

通常ログは常に有効とし、次を記録する。

- アプリの起動、二重起動拒否、終了
- 設定の作成、読込、保存、破損復旧
- スタート検出結果と監視状態
- ランチャーの状態遷移、表示・退出理由
- 表示とモーションの主要時間
- アクション、音声、トレイ操作の成否
- 例外種別、HRESULT、Win32エラー番号
- アプリ版、Windows版、ランタイム、プロセス構成、詳細ログ状態、モニター構成の環境情報

環境情報は起動時に必ず記録する。起動後はWindowsのディスプレイ変更、DPI変更、作業領域変更、スリープ復帰の通知を受け、750msのデバウンス後に再取得する。前回記録時とのフィンガープリント比較で差がある場合だけ追記する。トラブルシューティング表示時と診断設定適用時にも比較し、取りこぼした通知と詳細ログ状態の変更を補う。定期ポーリングとランチャー表示処理内での同期取得は行わない。

通常ログには次を直接記録しない。

- UI AutomationのAutomationId、候補一覧
- PID以外の詳細なウィンドウ情報、HWND、矩形
- 音声デバイスIDとデバイス名
- アクションの実行対象
- Windowsキーの低レベルなdown／up分類

## 4. 詳細診断ログ

設定画面の「トラブルシューティング」で有効にする。専用の適用操作を行った時点から24時間だけ有効で、期限後は自動的に詳細記録を停止する。一般設定の保存だけでは期限を延長しないことを設計上の正とする。

詳細ログには有効期間中の通常ログも複製する。そのため、調査時は詳細ログだけで全体の時系列を追える。通常ログへは詳細項目を逆流させない。

追加で記録する項目:

- UI Automationワーカーとフォーカス監視の開始・終了
- フォーカス要素とスタート候補のAutomationId、コントロール種別、矩形
- ランチャーの配置座標、サイズ、フォーカス状態
- 描画準備、Loaded、表示状態
- Windowsキーの単体／ショートカット分類
- アクション実行対象と詳細エラーメッセージ
- 音声切り替え先のデバイス名とID

## 5. パス匿名化

通常ログと詳細ログの両方で、ログキューへ入れる前に匿名化する。

既知のルートは次のトークンへ置き換える。

```text
アプリ配置フォルダー  → %APPDIR%
ローカルAppData       → %LOCALAPPDATA%
ローミングAppData     → %APPDATA%
一時フォルダー        → %TEMP%
ユーザープロファイル  → %USERPROFILE%
```

トークンより後ろのパス要素は `<redacted>` とする。別ユーザーの `C:\Users\名前` が例外メッセージに含まれた場合も、ユーザー名を `<user>` へ置き換える。改行とNULL文字は空白へ置き換え、ログ行の注入を防止する。

匿名化は共有前確認の代わりにはならない。UI Automation要素名は取得・記録しないが、コマンド、音声デバイス名、例外情報には個人情報が含まれる可能性があるため、ログを外部へ渡す前に利用者が内容を確認する。この注意はトラブルシューティング画面へ常時表示する。

## 6. 設定画面

別のトップレベルウィンドウは作らず、既存設定画面の下部からContentDialogを開く。

- 詳細診断ログのオン／オフ
- 現在状態と有効期限
- 通常・詳細ログの保存場所
- データフォルダーを開く
- OS、アーキテクチャ、.NET、アプリ版、モニター数、解像度、配置、作業領域、DPI、拡大率、取得可能なリフレッシュレートの環境情報をコピー
- 確認後にログを削除

詳細診断ログの切り替えは、ダイアログ内の「詳細診断ログ設定を適用」で確定する。項目編集に無効値や未保存変更があっても、診断機能だけを有効化できる。設定画面下部の「保存」を使った場合も現在の診断設定を保持する。

## 7. 実装上のルール

- 利用者向けの状態、成否、主要時間は `DiagnosticLogger.Write` を使う。
- 個人情報またはOS内部情報を含み得る調査値は `DiagnosticLogger.WriteDetailed` を使う。
- 呼び出し側だけの文字列置換へ依存せず、最終的な匿名化はlogger境界で必ず行う。
- 詳細ログが無効なとき、詳細メッセージはキューにもファイルにも入れない。
- ログ書込みはバックグラウンドライターで行い、UIスレッドから同期ファイルI/Oを行わない。
- 環境変更通知のハンドラーでは情報取得を行わず、UIスレッドの短いタイマーを再始動するだけにする。
- 新しいログ項目を追加した場合、通常／詳細のどちらに属するかと、共有時の情報漏えいリスクをレビューする。

## 8. 確認項目

1. 無効時に詳細ログへ新しい行が追加されない。
2. 有効化して保存すると通常イベントが両ファイルへ、詳細イベントが詳細ファイルだけへ入る。
3. 24時間経過後に詳細記録が停止する。
4. 再起動後も期限内だけ有効になる。
5. ログにWindowsユーザー名とユーザープロファイルの実パスが残らない。
6. ログ削除のキャンセルでは何も消えず、確定時に4ファイルが削除され、新しい通常ログへ現在の環境情報が記録される。詳細ログが有効な場合は新しい詳細ログにも記録される。
7. 非パッケージ版とMSIX版の両方で、`%LOCALAPPDATA%\Windows_SC`以下の表示中の保存先と実体が一致し、データフォルダーから`Settings`と`Logs`を確認できる。
8. ログと環境情報に不要な個人情報が含まれない。
9. 旧版の非パッケージ保存先、MSIX LocalCache、MSIX LocalStateから設定とログを失わず移行できる。
10. 起動時に環境情報が記録され、モニター構成等を変更した場合だけ新しい環境情報が追記される。同じ構成の通知やトラブルシューティング表示では重複しない。

## 9. 現行実装との差と残件

- 一般設定保存では既存の詳細診断期限を維持し、専用の適用操作だけが期限を更新する動作へ修正済み。
- 2MBのローテーション判定はロガー初期化時だけ行う。起動中に上限を超えたファイルは次回起動時にローテーションする。
- UI Automation要素名の取得・記録は廃止済み。AutomationId、コントロール種別、矩形は詳細ログだけに記録する。
- UIスレッド、AppDomain、未監視Taskの未処理例外は共通形式で通常ログへ記録する。致命的例外は処理済みにせず、通常の異常終了動作に従う。
